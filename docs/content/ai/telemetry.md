# 텔레메트리

프론트엔드 센서가 수집하는 마우스/키보드 행동 데이터를 두 채널로 분리해서 전송합니다. 실시간 검사에 필요한 압축 요약 수치는 HTTP 헤더로, 오프라인 AI 학습에 필요한 원본 좌표 데이터는 전용 API의 HTTP 바디로 전송합니다.

---

## 데이터 분리 전략

```
[ Frontend Sensor ] ─── (마우스/키보드 움직임 수집) ──> [ 데이터 포맷 분리 ]
                                                              │
                    ┌─────────────────────────────────────────┴──────────────────────────────────────────┐
                    ▼                                                                                    ▼
         [ Fast Path (헤더 전송) ]                                                         [ Bypass Path (바디 전송) ]
     페이로드: 요약된 핵심 수치 8종                                                       페이로드: 궤적(x,y) 원본 데이터 전체
     레이어: HTTP Header (x-defense-features)                                            레이어: 전용 수집 API의 HTTP Body
     목적: Proxy 파싱 지연 최소화, BE DTO 오염 방지                                        목적: 오프라인 대형 AI 패턴 학습용 데이터 축적
```

---

## 비교

| 항목 | Fast Path (실시간 동기용) | Bypass Path (사후 비동기용) |
|---|---|---|
| **데이터 사이즈** | 초경량 (직선성, 체류시간 등 8개 압축 수치) | 매우 무거움 (수백~수천 개의 원시 x, y 좌표 배열) |
| **전송 시점** | 기존 중요 API(결제 등) 호출 시 헤더에 기생(piggyback) 전송 | 주기적(5초 간격) 또는 화면 이탈 시 단독 전송 |
| **수신 주체** | Istio Envoy가 가로채어 AI 서버로 릴레이 | 텔레메트리 수집 전용 백엔드(S3/DB)로 전달 |

---

## 설계 이유

결제 API마다 수천 개의 좌표 배열을 바디에 실어 보내면 파싱 부하가 생기고, 백엔드 API 스펙이 AI 방어 로직에 오염됩니다.

이 문제를 해결하기 위해 프론트엔드에서 자체적으로 8개의 요약 수치를 계산한 뒤 HTTP 헤더(x-defense-features)에 실어 보냅니다. **백엔드는 이 사실을 몰라도 됩니다.** Envoy가 헤더를 가로채어 AI 서버로 릴레이하기 때문입니다.

무거운 원본 좌표 데이터는 결제와 무관한 전용 API를 통해 백그라운드로 전송되며, S3 Data Lake에 쌓여 오프라인 AI 학습에 활용됩니다.
