## 6.3 트러블슈팅 — 실행 중 마주친 문제와 대응

실제 운영에서 관찰·해결한 구체적 문제들이다.

### 1. 서버 큐 개방 타이밍은 정각에 맞지 않는다 (충돌 응답 409)

| 관찰 | 대응 |
|------|------|
| 오픈 정각에 정확히 클릭해도 서버 큐가 아직 안 열려 충돌 응답 반복. 백엔드 스케줄러가 초 단위로 정렬되지 않음 — 정각 뒤 수 초까지 큐 미개방 상태가 지속됨 | 500ms 고정 주기로 **최대 3분간 반복 클릭**, 큐가 열리는 순간 자동 진입. "정각에 한 번"이 아니라 "정각 이후 지속 시도"가 실제 공격의 현실 |

> **일반화**: 오픈 경쟁은 "정각 타이밍"이 전부가 아니라 **"서버가 실제로 열리는 순간까지 대기 가능한 정밀도"**의 싸움.

### 2. 한 PC에서 에이전트를 완전 정각에 동시 투입하면 호스트가 버벅댐

| 관찰 | 대응 |
|------|------|
| 단일 PC에서 5~10개 에이전트를 정각에 동시 시작하면 **호스트 리소스 경쟁**이 발생 — 여러 Chromium 프로세스가 CPU·메모리·네트워크를 동시에 점유하고, Playwright의 브라우저 통신 메시지도 단일 프로세스 안에서 직렬 처리되어 개별 에이전트 응답이 누적 지연됨 | 에이전트 순번별로 예매 버튼 클릭 시각에 **수 초씩 선형 지연** 주입. 호스트 부하가 시간 축으로 분산됨 |

#### "그럼 에이전트를 늦게 투입할 거면 스웜이 왜 유리한가?"

이 시간차는 **시뮬레이션 환경의 특수 조건을 보정**한 것이지, 실전에서는 불필요하다.

| 환경 | 예매 클릭 시간차 | 큐 순위 차이 | VQA 진입 시간차 |
|------|-------------|------------|-------------|
| **실전 (정상 유저 수만 명)** | 0.1초 | 수백~수천 번 | **1~2분** |
| **시뮬레이션 (경쟁 유저 부족)** | 0.1초 | 거의 동일 | 거의 동시 |

- **실전**: 에이전트의 예매 클릭 속도는 사람보다 월등히 빠르므로 정각 직후 상위권 진입은 거의 확정. 첫 진입이 100번대면 5번째도 5000번대 안쪽 — 전원이 **여전히 상위권**. 0.1초 차이만 있어도 큐 구조상 VQA 진입은 자연스럽게 분 단위로 벌어짐
- **시뮬**: 경쟁 유저가 없어서 예매 클릭을 동시에 하면 VQA도 동시에 뜨고, 호스트가 못 버팀. 그래서 **실전의 자연스러운 VQA 진입 시간차를 예매 클릭 시간차로 인위적으로 재현**

> **일반화**: 스웜의 본질은 "예매 클릭 타이밍의 동시성"이 아니라 "전원이 상위권 큐 순위를 확보하는 것". 시간차는 실전 환경과 시뮬 환경을 맞추기 위한 보정이지 약점이 아님.

### 3. VQA 드래그 착지 실패가 의외로 자주 일어남 (이번 실행 5건)

| 관찰 | 대응 |
|------|------|
| 글러브를 착지 마커까지 드래그할 때, 마우스 이동 곡선 합성 과정에서 수직 방향 떨림(tremor) 때문에 최종 위치가 목표에서 벗어나는 경우가 상당히 있음. 드롭했을 때 공간 판정에서 탈락 | 드래그 직후 **글러브 실제 위치를 재확인**하고 24px 이상 벗어나 있으면 현재 위치에서 재드래그(최대 3회). "이상적 곡선 합성"만으로는 불충분 |

> **일반화**: 자연스러움을 위해 넣은 노이즈(손떨림)가 역설적으로 **착지 정확도를 해친다**. 자연스러움과 정확성은 동시에 최적화되지 않음 → 사후 보정 장치 필요.

### 4. 장거리 이동 시 전 구간 인간 궤적은 너무 느리다

| 관찰 | 대응 |
|------|------|
| 구역·좌석 버튼까지 수백 픽셀 이동 시 전 구간 곡선 합성 + 자연스러운 속도로 움직이면 체감 지연이 누적. 경쟁 구간에서 속도 손실이 큼 | **숙련자 사전 위치 선점 모델** — 버튼이 화면에 나타나기 전부터 타깃 예상 위치 근처로 마우스를 미리 이동시키고, 실제 버튼이 뜨면 짧은 보정 이동으로 클릭 (e-sports 프로의 pre-aim과 같은 원리) |

> **일반화**: "인간처럼 움직인다 = 느리다"는 함정. 실제 **숙련된 인간의 행동 특성**을 정확히 모델링하면 속도와 자연스러움이 양립 가능.

### 5. 스웜 내 에이전트들이 같은 순간에 VQA를 풀면 위험

| 관찰 | 대응 |
|------|------|
| 한 스웜 안의 여러 에이전트가 같은 시간대에 챌린지를 풀면, 서로 독립 세션임에도 "비정상적으로 동기화된 풀이 패턴"으로 보일 수 있음 | 스웜 차원에서 **동시 풀이 수를 제한**(기본 1). 한 번에 한 에이전트만 풀고 나머지는 대기. 여기에 에이전트별 시작 지연까지 더해 완전히 동시에 시작되는 걸 방지 |

> **일반화**: 세션 격리를 했다고 해서 행동 패턴까지 독립적으로 보이는 건 아니다. **시간 축 상관관계**까지 끊어내야 의미 있는 분산.

### 6. LLM 코디네이터는 "자유롭게 판단"시키면 안 된다

| 관찰 | 대응 |
|------|------|
| 좌석 추천 모드 에이전트에게는 구역 지시가 무의미(시스템이 자동 배정)한데도 LLM이 지시를 내려는 경향 관찰. 자유로운 프롬프트는 **의미 없는 개입**을 유발 | **매우 보수적인 시스템 프롬프트** — "구역 변경은 명확한 차단 증거 있을 때만", "인원수 감소는 연석 못 찾았을 때만", "자동 배정 에이전트의 구역은 건드리지 말 것" 식으로 개입 조건을 엄격히 정의. 집계 단에서도 무의미한 지시는 자동 제외 |

> **일반화**: LLM의 창의성은 **잘못된 방향으로도 창의적**이다. 스웜 수준 자동화에서 LLM은 "제한된 결정권자"로 설계해야 안전.

### 7. 이번 실행의 구조적 관찰 — 방어는 앞단에 집중된다

이번 테스트에서 관찰된 차단·중단의 **단계별 분포**:

| 원인 단계 | 에이전트 수 | 특징 |
|---------|----------|------|
| 대기열에서 탈락 | 6 | VQA까지 도달 못함 |
| VQA 실패 → BLOCKED | 5 | 드래그 착지 실패·타이밍 실패 혼재 |
| 실행 중단 (ABORT) | 5 | 직접 좌석 선택 모드에서 상대적으로 집중 발생 |
| **좌석 선택 이후 차단** | **0** | Hold까지 도달한 20명 전원 성공, 에이전트 간 좌석 충돌도 0건 |

> **핵심 관찰**: **"방어는 앞단(대기열·VQA·구역 탐색)에서 선별적으로 작동한다"** — 좌석 선택 이후 단계에서는 차단이 일어나지 않음. 이는 **방어가 의심 세션을 뒤로 미루지 않고 앞단에서 끝내는 구조**임을 공격 로그만으로도 간접 확인한 것.

## 6.5 추후 개선 방향

## 공격 측

### 분산 실행 아키텍처 전환

**현재 구조의 근본 한계**

한 PC에 단일 에이전트만 돌리면 **지연 없이 최대 속도로 동작**한다. 그런데 같은 PC에 5~10개를 동시에 띄우면 Chromium 프로세스 간 CPU·메모리·네트워크 경합 + Playwright의 브라우저 통신 메시지 직렬 처리 때문에 **개별 에이전트 응답이 수 초씩 누적 지연**된다. 즉 현재 "스웜 당 한 PC" 구조는 호스트가 병목이다.

이 한계를 정면으로 해결하려면 **에이전트를 각자 독립 환경에 1:1로 배치하고, 중앙 코디네이터가 전략만 묶어서 관리**하는 분산 구조로 전환하는 것이 맞다.

**제안 구조**

```
[에이전트 환경 1]  ──┐
[에이전트 환경 2]  ──┤
[에이전트 환경 3]  ──┼──→  중앙 코디네이터 서버
    ...            ──┤     ├─ 이벤트 수신 (좌석 확보·차단·챌린지 결과 등)
[에이전트 환경 N]  ──┘     ├─ LLM 전략 판단
                           └─ 공유 전략 저장소 (Redis 등)
                                  ↓
                           에이전트로 전략 배포 (구역·인원수·제외 좌석)
```

| 구성 요소 | 역할 |
|----------|------|
| **에이전트 환경 (1:1)** | 각자 격리된 실행 공간 — 호스트 경합 완전 제거, 브라우저 하나만 돌아감 |
| **중앙 코디네이터** | 모든 에이전트 이벤트를 수집하고 LLM으로 스웜 전략 조정 |
| **공유 전략 저장소** | 에이전트가 구독·갱신하는 전략 상태 (Redis Pub/Sub 또는 메시지 브로커 기반) |

**실행 환경 격리 옵션**

| 방식 | 장점 | 단점 |
|------|------|------|
| **클라우드 VM 풀** (AWS EC2, GCP Compute 등) 에이전트당 1 VM | 완전 격리 + 탄력적 확장 + 지리적·IP 분산 자연스러움 | 비용 |
| **Kubernetes 클러스터** — 에이전트를 별도 노드에 배포 | 자동 스케일링·자동 복구, CI/CD 통합 | 운영 복잡도 |
| **물리 PC 여럿 묶기** — 지금 스웜 간 독립 구조의 자연 확장 | 비용 최소 + 실제 네트워크 경로 분산 | 수동 관리, PC 수 한계 |
| **하이브리드** — 물리 PC + 클라우드 혼합 | 유연성 | 운영 복잡도 ↑ |

**기대 효과**

| 효과 | 설명 |
|------|------|
| **호스트 경합 제거** | 에이전트가 각자 최대 속도로 동작 → 정각 동시 투입도 가능 |
| **실전 타이밍 재현** | 시뮬 환경에서도 "예매 클릭 시간차 인위 주입" 같은 보정 불필요. 자연스러운 큐 경쟁 재현 |
| **IP·네트워크 분산 강화** | 클라우드·여러 물리 환경으로 흩어져 현실 공격 상황(서로 다른 출발점)과 더 일치 |
| **LLM 코디네이터의 실질 활용** | 에이전트 개별 지연이 사라져 코디네이터 전략 지시가 즉각 반영 가능. 현재는 사후 이벤트 위주였던 트리거 분포가 선제 조정 쪽으로 확대될 여지 |
| **확장성** | 수십~수백 에이전트 규모도 무리 없이 실행 가능 |

**통신 프로토콜 후보**

| 옵션 | 특성 |
|------|------|
| Redis Pub/Sub | 단순·경량, 이미 방어팀 인프라에서 사용 중이라 친숙 |
| NATS | 분산 메시지 브로커, 낮은 지연 |
| gRPC 스트림 | 구조화된 양방향 통신, 스키마 강제 |
| WebSocket (커스텀) | 가장 간단, 방화벽 친화적 |

> **요약**: 현재 "1 PC = 1 스웜 = 다수 에이전트" 구조를 **"1 환경 = 1 에이전트, 다중 환경 = 1 스웜"** 으로 뒤집는 것. 호스트 경합을 설계 차원에서 제거하고, 중앙 LLM 코디네이터로 전체를 묶는 방식. 시뮬레이션 정확도·실전 근접성·확장성 세 축 모두에서 개선이 기대됨.
