# 4. AI Defense — 행동 기반 실시간 방어 시스템

## 4.1 AI Defense가 무엇인가

AI Defense는 티켓팅 예매 흐름에서 **자동화 봇과 비정상 행동을 실시간으로 탐지하고, 지연·검증·차단으로 대응하는 방어 시스템**이다.

이 시스템은 단순한 방화벽이나 요청 수 제한이 아니다.
같은 요청이라도 **그 세션이 지금까지 어떻게 행동했는지**, **보안 검증을 통과했는지**, **현재 어느 단계에 있는지**에 따라 다른 결정을 내린다.

정리하면 다음과 같다.

| 구분 | 의미 |
|------|------|
| **아님** | 단순 방화벽, 단순 Rate Limit |
| **맞음** | 행동 패턴 + 세션 상태를 함께 보는 위험도 기반 방어 |
| **핵심 특징** | 같은 API 요청도 세션 맥락에 따라 다른 판단 |

## 4.2 왜 두 평면(Plane)으로 나눴는가

방어 시스템은 두 가지 상충하는 요구를 동시에 만족해야 했다.

| 요구 | 의미 |
|------|------|
| **실시간성** | 요청 경로에서 빠르고 예측 가능해야 함 |
| **진화 대응력** | 공격 패턴 변화에 따라 정책이 바뀔 수 있어야 함 |

이 둘을 한 시스템 안에서 동시에 해결하려 하면, 실시간 판단은 무거워지고 정책 업데이트는 조심스러워진다.
그래서 방어를 **Online Plane**과 **Offline Plane**으로 분리했다.

| 평면 | 역할 | LLM 사용 | 설계 원칙 |
|------|------|----------|-----------|
| **Online Plane** | 실시간 요청 판단 | ❌ 사용 안 함 | 결정론, 재현성, 저지연 |
| **Offline Plane** | 로그 분석, 정책 개선, 사후 검토 | ✅ 제한적으로 사용 | 유연성, 자동화, 운영 개선 |

이 분리가 AI Defense 설계의 출발점이다.
이후의 거의 모든 구조적 선택은 이 원칙에서 나온다.

## 4.3 전체 요청 흐름과 주요 컴포넌트

```text
[사용자 브라우저]  마우스 행동 데이터 포함
      ↓
[서비스 메시 게이트웨이 (Envoy + Istio)]  중요 API만 선별
      ↓
[인증·방어 어댑터]  판단 요청 변환
      ↓
[AI Defense 런타임 (4단 파이프라인)]
      ↓ ↕
[Redis]  세션 상태·차단 상태·정책 캐시

         ─── Offline Plane ───

[런타임 감사 로그] → [S3 아카이브] → [ETL] → [ClickHouse 분석 저장소]
                                                ├→ [정책 자동 최적화기] → [PostgreSQL 정책 권위] → [Redis 재동기화]
                                                └→ [사후 검토 도우미] → [의심 세션만 백엔드 전달]
```

이 구조에서 핵심 선택은 세 가지다.

| 선택 | 이유 |
|------|------|
| 중요 API만 검사 | 모든 요청을 검사하면 지연과 오탐 영향이 커짐. 큐·좌석·챌린지·결제 인접 구간만 본다. |
| 어댑터를 중간에 둠 | AI Defense는 방어 개념으로 판단하고, 어댑터가 그것을 실제 HTTP 응답과 헤더로 번역한다. |
| 장애 시 통과(fail-open) | AI Defense 장애가 서비스 전체 중단으로 번지면 안 된다. 방어력 저하는 감수하되 서비스는 유지한다. |

## 4.4 Runtime 판단의 4단 파이프라인

런타임 판단은 아래 네 단계로 나뉜다.

```
1. 점수 계산  →  2. 증거 축적  →  3. 정책 적용  →  4. 실행·상태 반영
```

각 단계의 역할은 명확히 분리되어 있다.

| 단계 | 입력 | 출력 | 책임 |
|------|------|------|------|
| 1. 점수 계산 | 행동 지표 + 외부 검증 점수 | 위험 점수·위험 등급 | 숫자 계산 |
| 2. 증거 축적 | 이벤트 스트림 | 파생 신호·세션 카운터 | 규칙 기반 패턴 감지 |
| 3. 정책 적용 | 점수 + 증거 + 현재 상태 | 다음 행동 결정 | 우선순위 규칙 적용 |
| 4. 실행·상태 반영 | 행동 결정 | HTTP 응답 + 세션 상태 저장 | 차단·챌린지·전이 검증 |

이렇게 나눈 이유도 분명하다.

- 계산과 실행이 한곳에 섞이면 디버깅이 어려워진다.
- 단계별 로그를 남기면 어느 단계에서 판단이 달라졌는지 바로 볼 수 있다.
- 상태 쓰기 책임을 분리하면 세션 상태 오염을 줄일 수 있다.

실제로 각 단계가 쓸 수 있는 상태도 구분되어 있다.

| 단계 | 쓸 수 있는 상태 |
|------|----------------|
| 점수 계산 | 위험 점수, 등급 |
| 증거 축적 | 카운터, 연속 실패 기록 |
| 실행 | 흐름 상태, 검증 통과 여부, 차단 상태 |

## 4.5 위험 점수 계산 — 행동을 숫자로 바꾸는 방식

위험 점수는 다섯 가지 행동 지표를 바탕으로 계산한다.

| 지표 | 범위 | 위험 신호로 해석되는 방향 |
|------|------|--------------------------|
| 손떨림 표준편차 | 0 ~ 6 | 너무 낮으면 기계적 |
| 직선성 비율 | 0 ~ 1 | 1에 가까울수록 과도하게 직선적 |
| 평균 속도 | 0 ~ 4000 | 너무 빠르면 비정상 |
| 클릭 전 머뭇거림 | 0 ~ 2000 | 너무 낮으면 인간다운 망설임 부족 |
| 경로 비율 | 1 ~ 3 | 1에 가까울수록 완벽한 직선 이동 |

이 지표들을 가중 합해 내부 점수를 만든다.

```
s_내부 = 0.35·떨림_bot + 0.25·직선성_bot + 0.15·속도_bot
       + 0.15·머뭇거림_bot + 0.10·완벽한직선
```

가중치의 의도는 다음과 같다.

| 지표 | 가중치 | 의도 |
|------|--------|------|
| 손떨림 | 0.35 | 인간의 자연스러운 떨림은 가장 흉내내기 어려운 축 |
| 직선성 | 0.25 | 두 번째로 중요한 판별 축 |
| 속도 | 0.15 | 빠른 자동화 패턴 포착 |
| 머뭇거림 | 0.15 | 즉시 반응하는 기계성 포착 |
| 완벽한 직선 플래그 | 0.10 | 극단적 조합 보조 포착 |

여기에 외부 검증 점수도 결합한다.

```
s_외부봇도 = 1 − 외부점수
s_현재 = 0.30·s_외부봇도 + 0.70·s_내부
```

내부 신호 비중을 더 크게 둔 이유는, 외부 점수는 선택적이거나 타임아웃될 수 있지만 내부 행동 계산은 항상 가능하기 때문이다.

단일 이벤트만 보지 않기 위해 점수는 지수가중이동평균(EWMA)으로 누적한다.

```
r_새 = (1 − α)·r_이전 + α·s_현재     (α = 0.30)
```

이 방식은 한 번의 좋은 행동으로 위험 점수가 급락하는 것을 막고, 반대로 한 번의 튀는 행동만으로 즉시 정상 사용자 전체를 의심하지 않도록 완충해 준다.

추가로 두 가지 감쇠가 들어간다.

| 감쇠 | 조건 | 효과 |
|------|------|------|
| 비활성 감쇠 | 일정 시간 비활동 | 오래 쉰 세션의 위험 점수 완화 |
| 검증 통과 감쇠 | 챌린지 통과 | 위험 점수 절반 감소 |

다만 챌린지 통과 시에도 완전 초기화는 하지 않는다.
한 번 통과했다고 곧바로 정상 사용자로 단정할 수는 없기 때문이다.

## 4.6 위험 등급과 기본 행동

누적된 위험 점수는 네 개의 등급으로 변환된다.

| 등급 | 점수 범위 | 기본 행동 | 의미 |
|------|-----------|-----------|------|
| T0 | < 0.20 | 그대로 통과 | 정상 |
| T1 | < 0.50 | 약한 지연 | 경미한 의심 |
| T2 | < 0.80 | 강한 지연 | 높은 의심 |
| T3 | ≥ 0.80 | 차단 | 고위험 |

등급 경계에서 위아래로 계속 흔들리는 문제를 막기 위해 두 가지 장치를 넣었다.

| 장치 | 원리 | 해결 문제 |
|------|------|-----------|
| 히스테리시스 | 내려갈 때만 추가 여유를 둠 | 경계값 진동 |
| 보호 대기 | 챌린지 통과 직후 일정 시간 다운그레이드 억제 | "통과 직후 재공격" 패턴 |

실제 행동 결정은 단순히 등급만으로 끝나지 않는다.
우선순위 규칙이 먼저 적용된다.

1. 이미 차단 중이면 계속 차단
2. 종료된 세션이면 새 개입 없음
3. 구역/좌석 단계에서 챌린지 미통과면 챌린지 강제
4. 결제 단계면 새 마찰 없음
5. 그 외에는 등급별 기본 행동 적용

즉 등급은 기본 방향을 주고, 실제 결정은 상태와 단계까지 포함해 내려진다.

## 4.7 규칙 기반 증거 축적 — 단시간 패턴 감지

점수 계산이 행동의 모양을 본다면, 증거 축적은 짧은 시간 안의 반복 패턴을 본다.

| 패턴 | 감지 조건 | 의미 |
|------|-----------|------|
| 단시간 고가치 클릭 폭주 | 약 1.5초 내 좌석 클릭/확보 3회 이상 | 대량 선점 시도 |
| 과도한 조회 스캐닝 | 약 2초 내 조회 API 8회 이상 | 비정상 탐색 |
| 챌린지 연속 실패 | 60초 안에 챌린지 2회 이상 실패 | 사람 사용자와 다른 실패 패턴 |

이 신호들은 점수와 별개로 작동한다.
즉 점수는 아주 높지 않더라도, 짧은 구간에서 비정상 반복이 보이면 정책 결정 이유에 플래그로 붙는다.
이 덕분에 "모양은 그럴듯하지만 반복성은 비정상적인" 공격도 잡아낼 수 있다.

## 4.8 VQA(보안 챌린지) — 2중 게이트

이 챌린지의 핵심은 다음 한 문장이다.

> **정답을 맞혔다고 해서 사람이 되는 것은 아니다.**

자동화 공격은 정답 좌표와 타이밍을 계산할 수 있다.
그래서 AI Defense는 챌린지를 정답 검증과 과정 검증의 2층 구조로 설계했다.

| 게이트 | 검증 내용 | 대상 |
|--------|-----------|------|
| 1층: 공간·시간 검증 | 올바른 위치에서, 올바른 순간에 잡았는가 | 정답 |
| 2층: 행동 궤적 검증 | 그 과정을 사람처럼 풀었는가 | 풀이 과정 |

2층 이상 판정에 쓰는 기준은 아래와 같다.

| 지표 | 이상 기준 |
|------|-----------|
| 직선성 | ≥ 0.985 |
| 경로 비율 | ≤ 1.03 |
| 떨림 | ≤ 0.20 |
| 속도 | ≥ 2200 |
| 머뭇거림 | ≤ 80 |

이 값들은 "정답은 맞혔지만 푸는 과정은 지나치게 기계적"인 경우를 겨냥한다.
정상 사용자가 이 조합을 지속적으로 만족시키는 것은 매우 어렵다.

공격자 입장에서 이 챌린지가 까다로운 이유도 여기 있다.

- 1층을 뚫으려면 정답 좌표와 타이밍을 계산해야 한다.
- 2층을 뚫으려면 그 정답을 사람처럼 풀어내는 마우스 움직임 자체를 만들어야 한다.

연속 실패가 누적되면 재시도 제한과 임시 잠금도 걸린다.
즉 챌린지는 단순한 퀴즈가 아니라, 정답 + 과정 + 반복 실패 관리를 모두 포함한 방어 계층이다.

## 4.9 오프라인 정책 자동 최적화기

여기서 LLM은 실시간 자동 적용기가 아니다.
역할은 어디까지나 **정책 제안 도구**다.

작동 흐름은 다음과 같다.

```
① 분석 저장소에서 지표 스냅샷 조회
    ↓
② 실행 조건 확인 (이벤트 수·세션 수·쿨다운)
    ↓
③ 규칙 기반 제안 우선
    ↓
④ 규칙으로 충분하지 않을 때만 LLM 제안
    ↓
⑤ 검증기 통과 여부 확인
    ↓
⑥ 카나리 방식 단계적 배포
```

규칙 기반 제안 예시는 아래와 같다.

| 조건 | 제안 |
|------|------|
| 평균 지연이 과도하게 높음 | 강한 지연을 소폭 감소 |
| 차단율과 챌린지 실패율이 함께 높음 | 점수 누적 계수를 소폭 완화 |

하지만 변경 가능한 범위는 엄격히 제한된다.

| 수정 가능 | 수정 불가 |
|-----------|-----------|
| 점수 누적 계수 | 챌린지 재시도 횟수 |
| 등급 임계값 | 챌린지 잠금 시간 |
| 히스테리시스 여유 | 차단 유지 시간 |
| 보호 대기 시간 | 보안 게이트 활성 여부 |
| 지연 값 | |

이 허용 목록이 중요한 이유는, LLM이 어떤 제안을 하더라도 건드리면 안 되는 사용자 경험 직결 파라미터는 애초에 수정 대상에서 제외되기 때문이다.

배포도 바로 100% 적용하지 않는다.

```
5% → 20% → 50% → 100%
각 단계에서 지표 감시 → 악화 시 자동 롤백
```

이 과정을 통해 실험적 변경이 전체 서비스에 즉시 전파되는 위험을 줄인다.

## 4.10 사후 검토 워크플로우 — Human-in-the-Loop

온라인 방어는 의도적으로 완충적으로 설계되어 있다.
즉 T1·T2 같은 중간 등급 세션은 즉시 차단하지 않고 지연이나 챌린지로 처리한다.
문제는 이 회색지대 세션들이다. 그래서 별도의 사후 검토 레이어를 둔다.

워크플로우는 여섯 단계다.

| 단계 | 내용 |
|------|------|
| 1. 후보 수집 | 분석 저장소에서 세션 목록 조회 |
| 2. 후보 필터링 | T1/T2 흔적이 있지만 즉시 차단되지는 않은 세션 선별 |
| 3. 세션 분석 | 타임라인 요약과 의심 신호 생성 |
| 4. 리뷰 판단 | LLM 또는 규칙 기반 폴백 |
| 5. 실행 요약 | 배치 단위 요약 생성 |
| 6. 저장·전달 | 권위 저장소 기록 + 의심 세션만 백엔드 전달 |

LLM이 없어도 동작하도록 폴백도 준비했다.

> T2 도달, 챌린지 실패, 지연 적용, 최근 행동이 지연 중
> 이 중 하나라도 있으면 의심 후보로 본다.

이 폴백이 중요한 이유는 두 가지다.

| 이유 | 설명 |
|------|------|
| 운영 연속성 | LLM 장애나 API 키 문제와 무관하게 워크플로우가 돌아가야 한다. |
| 역할 분리 | LLM은 정확도 향상 도구이지, 시스템의 필수 조건이 아니다. |

또한 백엔드에는 의심 세션만 전달한다.
정상 세션까지 보내면 불필요한 후처리가 늘고, 잘못된 제재 위험도 커지기 때문이다.

## 4.11 정책 권위 저장소와 런타임 캐시의 분리

정책은 두 계층으로 나뉜다.

| 계층 | 저장소 | 역할 | 주 사용자 |
|------|--------|------|-----------|
| 권위 계층 | PostgreSQL | 정책 원본, 버전, 롤아웃 상태 | 오프라인 도구 |
| 서빙 계층 | Redis | 런타임용 정책 문서와 캐시 | 런타임 |

핵심 원칙은 명확하다.

> **런타임은 PostgreSQL을 직접 읽지 않는다.**
> 요청 경로에서는 Redis에 있는 서빙용 정책만 읽는다.

이렇게 분리한 이유는 세 가지다.

| 관점 | 이유 |
|------|------|
| 성능 | 요청 경로에서 관계형 DB를 직접 읽으면 누적 지연이 커진다. |
| 안정성 | 권위 저장소가 느리거나 유지보수 중이어도 런타임은 계속 돌아야 한다. |
| 감사 추적 | 정책 변경의 원천은 한 곳이어야 하고, 서빙 캐시는 언제든 재생성 가능해야 한다. |

캐시에는 마지막 갱신 시각이 함께 기록된다.
허용치를 넘긴 오래된 캐시는 오류로 취급한다. 엄격 모드에서는 누락, 디코딩 실패, 신선도 만료 모두를 명시적 오류로 드러낸다.
이유는 간단하다. "오래된 정책으로 운영 중인데 아무도 모르는 상태"가 가장 위험하기 때문이다.

## 4.12 감사 로그 적재와 분석 구조

런타임의 모든 판단은 구조화된 로그로 남긴다.
공통 필드에는 시간, 세션 식별자, 이벤트 종류, 흐름 상태, 위험 등급, 결정 행동, 이유 코드, 정책 버전 등이 들어간다.

필수 이벤트는 대체로 다음 범주를 포함한다.

- 점수 계산
- 증거 축적
- 정책 결정
- 실행 완료
- 지연 적용
- 차단 실행
- 챌린지 결과
- 외부 검증 결과

로그는 세 층으로 저장된다.

| 저장소 | 역할 | 특징 |
|--------|------|------|
| 구조화 로그 파일 | 실시간 기록 | append-only |
| S3 | 장기 보관, 재처리 원천 | 영구 아카이브 |
| ClickHouse | 집계·질의·대시보드 | 분석용 저장소 |

이 세 층은 용도가 다르다.
로그 파일은 기록, S3는 원본 보존, ClickHouse는 분석에 최적화되어 있다.
이 위에 세션별, 경기별, 사후 검토 후보별 뷰를 올려 오프라인 도구와 대시보드가 공통 입력으로 사용한다.

## 4.13 배포 자동화 — 핵심은 순서 보장

런타임이 제대로 뜨려면 사전 작업이 먼저 끝나 있어야 한다.
배포 자동화는 그 순서를 강제한다.

| 순서 | 단계 | 내용 |
|------|------|------|
| 1 | 저장소 스키마 적용 | 권위 저장소 테이블 최신화 |
| 2 | 기본 정책 시드 | 초기 정책과 롤아웃 상태 준비 |
| 3 | 런타임 캐시 재동기화 | 권위 저장소 → Redis 반영 |

이 중 하나라도 실패하면 런타임은 올라가지 않는다.
의도는 단순하다. 미완성 상태의 런타임이 서비스에 붙는 사고를 막기 위해서다.

또 각 단계는 재실행 가능하게 설계했다.

- 이미 있는 스키마는 건너뛴다.
- 기본 정책이 있으면 덮어쓰지 않는다.
- 캐시 재동기화는 같은 내용을 다시 써도 된다.

즉 실패 후 같은 절차를 다시 돌려도 안전하다.

## 4.14 실패 모델과 복구 전략

실패 유형마다 원인과 복구 경로를 분리해 두었다.

| 실패 유형 | 원인 | 복구 방법 |
|-----------|------|-----------|
| 정책 읽기 실패 | 캐시 누락, 디코딩 오류, 신선도 만료 | 권위 저장소에서 Redis 재동기화 |
| 권위 저장소 쓰기 실패 | DB 오류 | 캐시 동기화 중단, 먼저 권위 저장소 복구 |
| 캐시 쓰기 실패 | Redis 오류 | 재시도 및 재동기화 |
| 분석 저장소 적재 실패 | ETL 오류 | S3 원천에서 재처리 |
| 백엔드 전달 실패 | 네트워크 또는 외부 시스템 오류 | 실패 상태 기록, 후속 재처리 |

복구의 원천도 구분되어 있다.

| 목적 | 원천 |
|------|------|
| 정책의 현재 상태 | PostgreSQL |
| 런타임 서빙 | Redis |
| 관측 재처리 | S3 + ClickHouse |

여기서 중요한 원칙이 하나 있다.

> 런타임 장애를 빨리 고치겠다고 PostgreSQL을 요청 경로에 직접 연결하면 안 된다.
> 그건 복구가 아니라 설계 위반이다.

## 4.15 현재 구조의 장점, 제약, 확장 포인트

### 장점

| 측면 | 내용 |
|------|------|
| 예측 가능성 | 온라인 판단이 빠르고 결정론적이다. |
| 디버깅 용이성 | 4단 분리로 어느 단계에서 달라졌는지 추적 가능하다. |
| 감사 추적성 | 권위와 서빙을 분리해 이력과 운영을 동시에 잡는다. |
| 재현 가능 계산 | EWMA, 감쇠, 히스테리시스, 고정 임계값이 모두 단순하고 설명 가능하다. |
| 자동화 위험 최소화 | 최적화기가 건드릴 수 있는 범위를 제한했다. |
| 운영 연속성 | 사후 검토가 LLM 없이도 돌아간다. |

### 제약

| 제약 | 영향 |
|------|------|
| 챌린지 텔레메트리 의존 | 프론트엔드 SDK가 불안정하면 정상 사용자도 실패할 수 있다. |
| 경기 식별자 추출 모호성 | URL·세션 형식에 의존하는 케이스가 남아 있다. |
| fail-open의 방어 공백 | 장애 시 서비스는 유지되지만 방어력은 낮아진다. |
| 외부 검증 폴백 반복 | 외부 신호 정보량이 줄어들 수 있다. |
| 캐시 신선도 트레이드오프 | 너무 느슨하면 오래된 정책을 오래 쓴다. |

### 확장 포인트

1. 챌린지 텔레메트리 품질과 클라이언트 SDK 안정성 강화
2. 경기 식별자 추출을 백엔드 계약 기반으로 정리
3. fail-open 발생률과 런타임 지연에 대한 별도 경보 체계 강화
4. 최적화기 자동 적용 범위를 충분한 데이터 축적 후 점진적으로 확대
