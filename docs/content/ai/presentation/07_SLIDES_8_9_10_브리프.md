## 장표 8 — 공격자가 되어본다

### 한 줄 메시지
> "시나리오 방어는 시나리오 밖에서 뚫린다. 그래서 우리는 **스스로 공격자가 되었다.**"

### 슬라이드 담을 것

**① 7장에서 이어받는 문제 제기**
- T0~T3(Rate Limit · Fingerprint · IP 평판)는 모두 **"많은 요청"**이라는 공통점을 가정
- T3+는 반대로 움직인다 — 소수 요청 · 인간 마우스 궤적 모방 · dwell time 랜덤 · LLM으로 좌석 고민
- → 기존 방어 전부 무효

**② 우리의 선택: 공격과 방어를 동시에 만든다**

| 레포 | 역할 | 스택 |
|------|------|------|
| `203-goormgb-ai-attack` | 휴먼 모방 AI 공격 에이전트 | LangGraph + Playwright |
| `201-goormgb-ai` | AI Defense (Online D0 + Offline LLM) | D0 런타임 + 오프라인 튜닝 |

**③ 공격 에이전트 핵심 (짧게)**
- **형태는 인간, 속도는 장인** — 마우스 궤적·떨림까지 재현
- 보안 퀴즈도 사람처럼 드래그로 풀기
- **에이전트=결정론 · 스웜 전략=LLM (비동기)** — 실행은 재현 가능하게, 전략만 학습형

**④ 9장으로 전환 (한 줄 브리지)**

> "이 정도 공격을 실제로 받아낸 방어는 어떻게 설계했는가 —"

단순 전환만. 아키텍처 원칙 공개는 9장에서.

## 장표 9 — AI Defense 아키텍처 (1장 권장 · 2장까지 허용 · ~3분)

### 한 줄 메시지
> **온라인은 빠르게(D0 결정적), 오프라인은 정확하게(LLM).**
> 두 경로를 분리해서 지연도, 적응력도 둘 다 확보.

### 슬라이드 담을 것

**① Online Plane — 결정적 D0 런타임 (LLM 없음)**
- **ext_authz 경로**: Istio가 중요 API만 AI Defense API로 forward
  - `/ai/precheck(QUEUE_ENTER)` · `/ai/evaluate(SEAT_ENTRY, ASSIGN_HOLD)`
- **행동 지표 7가지**: `linearityRatio`, `tremorStdDev`, `dwellTime`, `pathRatio`, `avgVelocity`, `totalDist`, `botRisk`
- **대응**:
  - 등급 T0 통과 / T1 약한 지연 / T2 강한 지연 / T3 차단
  - 좌석 선택 단계 진입 시엔 등급 무관하게 **VQA 관문 = `REQUIRE_S3`** (결정적 게이트, HTTP 428)
- **원칙**: 자연어 판단 없음, 결정론적 규칙 — 요청 경로 지연 최소화

**② Offline Plane — 사후 LLM 판단 (두 갈래 병렬 트랙)**

감사 로그 파이프라인: runtime audit → **S3 archive → ETL → ClickHouse/PostgreSQL**

그 위에 LLM 기반 두 트랙이 병렬로 돎:

| 트랙 | 컴포넌트 | 입력 | LLM 판단 | 결과 |
|------|----------|------|----------|------|
| **A. 세션 단위 재검토** | `backoffice_copilot` | 회색지대 세션(T1·T2, 차단 안 된 세션)만 필터 | 세션별 `NORMAL` / `SUSPICIOUS` | PostgreSQL `post_review_session_results` + 백엔드 전달 |
| **B. 정책 파라미터 튜닝** | `OfflineOptimizer` | 집계 지표 (ClickHouse) | 파라미터 조정 제안 | 5%→20%→50%→100% 단계 배포, 악화 시 자동 롤백 |

- 둘 다 **LLM 장애 시 규칙 기반 폴백**으로 계속 동작 → 운영 연속성
- **감사·재현성**: audit log + policy version 이력 → 매크로 분쟁 시 증거 보존

## 장표 10 — 맞대결 결과 + 마무리

### 한 줄 메시지
> "**알려진 취약점**을 찾는 침투테스트를 넘어, 우리는 **알려지지 않은 공격**까지 재현해 맞대결했다."

### 슬라이드 담을 것

**① 실측 KPI (미확정 — 수치 들어오면 반영)**

| 지표 | 의미 | 비고 |
|------|------|------|
| 자체 AI 공격 에이전트 탐지율 | T3+ 공격을 얼마나 잡았는가 | 공격/방어 같은 팀의 정직한 지표 |
| False Positive 비율 | 정상 사용자 오탐 | 너무 높으면 방어의 실용성 하락 |
| 판정 지연 p95 | D0 런타임 판정 latency | ext_authz 경로의 SLA |

**② 양쪽이 같은 구조에 도달했다 (마무리 시너지)**

8·9장을 다 들은 청중에게 "공격과 방어를 각자 만들어보니, 아키텍처 원칙이 같은 곳으로 수렴했다"는 깨달음을 제공. 여기서야 비로소 아래 표가 설득력을 가짐.

|  | 결정론 (빠름 · 재현) | LLM (학습 · 유연) |
|--|---|---|
| **공격** | Agent 워크플로우 | Swarm Coordinator |
| **방어** | D0 런타임 (Online) | 오프라인 튜닝·재검토 (Offline) |

> LLM을 런타임에 두면 지연이, 아예 빼면 적응력이 없다.
> 그래서 양쪽 모두 **실행은 결정론, 판단만 LLM** 으로 분리했다.

**③ 클로징 훅**
> 침투테스트는 **알려진 취약점**을 찾고,
> 우리는 **알려지지 않은 공격**까지 재현했습니다.
> 티켓팅 매크로 방어는 보안 테스트 한 번이 아니라,
> **공격자와 같은 속도로 진화하는 시스템**이 답이라고 생각합니다.

**④ 11장 연결**
- "그리고 궁극적으로는 — 매크로의 **경제성 자체를 깎는다**"
- → 11장(추천 분산 설계)으로 패스
