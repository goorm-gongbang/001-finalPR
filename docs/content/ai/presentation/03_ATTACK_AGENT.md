# 3. 공격자 심화 설명 (AI Attack Agent)

## 3.1 블랙박스 전제 — 공격자도 방어 내부를 모른다

공격 에이전트는 방어 내부 로직을 모른다는 전제에서 설계했다.
실제 공격자도 서버 내부의 feature 계산, 임계값, 상태 전이 규칙을 직접 볼 수는 없기 때문이다. 방어 구현을 미리 참고한 상태에서 우회 로직을 만들면, 그건 현실적인 공격 검증이 아니라 정답을 알고 푸는 테스트에 가까워진다.

개발은 아래 사이클의 반복이었다.

```
가설 수립 → 우회 설계 → 실행 관찰 → 원인 추정 → 다시 가설
```

예를 들어 "마우스 궤적 통계를 볼 것이다", "클릭 빈도를 볼 것이다", "같은 세션의 반복 실패를 볼 것이다" 같은 가설을 세운 뒤 실제 실행 결과로 확인했다.

초기 시행착오도 이 과정에서 나왔다.

| 초기 시도 | 관찰 | 수정 |
|-----------|------|------|
| 보안 챌린지 구간만 인간형, 나머지는 빠른 클릭 | 혼합 프로파일이 오히려 더 쉽게 탐지 | 전 구간 인간형으로 확장 |
| 고정 상수 궤적 | 패턴 학습에 취약 | 매 클릭마다 샘플링 + 가우시안 변동 주입 |
| 단순 sleep으로 오픈 시각 대기 | OS 스케줄러 편향으로 수십~수백 ms 오차 | 두 단계 대기 (느긋한 sleep + 정밀 폴링 루프) |

## 3.2 시스템 구조 — 세 개의 계층

공격 시스템은 크게 에이전트, 스웜, 시뮬레이션의 세 계층으로 나뉜다.

| 계층 | 단위 | 원칙 | LLM 사용 |
|------|------|------|----------|
| 에이전트 | 단일 공격 주체 | 결정론적 재현 | ❌ |
| 스웜 | 한 PC 안의 에이전트 5~10개 + 코디네이터 | 유연한 전략 적응 | ✅ (스웜 전략만) |
| 시뮬레이션 | 여러 PC가 각자의 스웜을 동시 실행 | 현실적 분산 공격 재현 | ❌ (PC 간 통신 없음) |

```
[시뮬레이션 계층] 여러 PC가 각자의 스웜을 돌림 — PC 간 통신 없음
         ↑ 결과 집계
[스웜 계층]      한 PC 안의 에이전트 5~10개 동시 실행 + LLM 코디네이터
         ↑ 전략 공유
[에이전트 계층]  상태머신 기반 결정론 워크플로우 (예매 1회 수행)
```

계층을 나눈 이유는 요구가 다르기 때문이다.
에이전트는 재현성과 디버깅이 중요하므로 결정론적으로 유지했다.
스웜은 방어 반응에 따라 전략을 바꿔야 하므로 LLM을 허용했다.
시뮬레이션은 여러 독립 공격원이 동시에 들어오는 현실적 조건을 재현하는 것이 목적이므로, PC 간 통신 없이 각자 실행하도록 설계했다.

## 3.3 에이전트 워크플로우 — 상태머신

단일 에이전트는 예매 한 번을 수행하는 상태머신으로 구현했다.

```
초기화 → 예매 진입 → 대기열 → 구역 선택 → 좌석 선택 → 종료
                  ↑
         (보안 챌린지는 어느 단계에서든 인터럽트로 발생 가능)
```

각 단계의 역할은 명확하다.
초기화 단계에서는 브라우저를 띄우고 저장된 세션으로 로그인 상태를 복원한다.
예매 진입 단계에서는 목표 시각에 맞춰 예매 버튼을 클릭하고 대기열로 들어간다.
대기열에서는 큐 통과를 기다리면서 보안 챌린지 등장 여부를 감시한다.
이후 구역 선택과 좌석 선택 단계를 거쳐 좌석 확보에 성공하거나, 차단·중단 상태로 종료된다.

이 구조를 상태머신으로 만든 이유는 세 가지다.
- 어느 단계에서 막혔는지 원인을 단계 수준으로 좁힐 수 있다.
- 보안 챌린지가 중간에 끼어들어도 "상태 저장 → 챌린지 처리 → 원래 상태 복귀" 패턴으로 일관되게 처리할 수 있다.
- 단계 간 분기 로직을 한곳에 모아 흐름을 예측 가능하게 유지할 수 있다.

성공 기준을 결제 완료가 아니라 좌석 확보까지로 둔 것도 의도적이다.
좌석이 확보된 시점부터는 점유 목적이 달성된 상태이고, 실제 결제는 PG사 연동 때문에 사람 개입과 대기 시간이 커져 자동화 루프의 품질을 평가하기에 부적합했다.

## 3.4 오픈 시각 동기화

예매 오픈 시각에 맞춰 진입하는 문제는 생각보다 까다로웠다.
단순히 sleep 한 번으로 기다리면 OS 스케줄러 편향 때문에 수십~수백 ms 오차가 발생했고, 이는 티켓팅처럼 경쟁이 극단적인 환경에서는 치명적이었다.

그래서 대기 방식은 두 단계로 나눴다.

| 구간 | 방식 | 정확도 | 목적 |
|------|------|--------|------|
| 목표 시각 5초 전까지 | 일반 sleep | ±수십 ms | 리소스 절약 |
| 마지막 5초 | 1ms 주기 타이트 루프 | ±수 ms | 정각 진입 정확도 확보 |

이중 구조의 목적은 분명하다.
초반 구간은 CPU 낭비를 줄이고, 마지막 짧은 구간만 정밀하게 폴링해 정확도를 확보하는 것이다.

스웜 차원에서는 모든 에이전트가 같은 순간에 클릭하지 않도록 분산도 넣었다.
정각 동시 클릭은 시간 상관관계가 너무 강해 봇 집단으로 보이기 쉽기 때문이다.
그래서 에이전트별로 선형 지연을 주어 시간축으로 퍼뜨렸고, 서버가 아직 큐를 열지 않아 409를 반환하면 일정 간격으로 반복 재시도하게 했다.

## 3.5 마우스 궤적 — "형태는 인간, 속도는 장인"

이 공격자의 핵심은 형태와 속도를 분리한 데 있다.

| 속성 | 전략 | 이유 |
|------|------|------|
| 형태 | 인간 샘플 분포에서 샘플링 + 가우시안 변동 | 동일 시그니처 방지, 통계 기반 탐지 회피 |
| 속도 | "티켓팅 장인" 페르소나 고정값 | 인간 상위 극단 속도로 경쟁력 확보 |

여기서 페르소나는 "어디를 클릭할지 이미 알고 있고, 망설임 없이 곧바로 움직이는 숙련자"다.

평균 인간을 그대로 복제하면 티켓팅 경쟁에서 너무 느리다.
반대로 한 사람의 패턴을 완전히 고정하면 시그니처가 반복되어 탐지에 취약하다.
그래서 형태는 다양하게, 속도는 빠르게 고정하는 방식을 택했다.

구체적인 궤적 생성은 다음 6단계로 이루어진다.

| 단계 | 역할 |
|------|------|
| 1. 샘플링 | 인간 궤적 샘플 중 하나를 선택해 기본 캐릭터 결정 |
| 2. 미세 변동 주입 | 직선성·떨림에 가우시안 변동 추가 |
| 3. 속도 덮어쓰기 | 원본 속도 정보는 버리고 숙련자 페르소나 값으로 교체 |
| 4. 곡선 생성 | 3차 베지어 곡선으로 자연스러운 경로 구성 |
| 5. 떨림 주입 | 경로 수직 방향에만 노이즈를 주어 모양만 흔들기 |
| 6. 역계산 검증 | 생성 결과를 다시 측정해 목표 분포와 비교·기록 |

중요한 점은 떨림을 진행 방향이 아니라 수직 방향으로만 준다는 것이다.
진행 방향으로 흔들면 타이밍과 착지점 정확도가 무너지고, 수직 방향으로 흔들면 적중률은 유지하면서도 경로 모양만 사람처럼 흐트러뜨릴 수 있다.

샘플 수집도 별도로 진행했다.
브라우저에 스크립트를 주입해 실제 사용자 마우스 이벤트를 수집했고, 다양한 거리와 방향을 포함하도록 50개 이상 샘플을 확보했다.
또 수집 시에는 일부러 "숙련자처럼 빠르고 직선적으로 클릭하는 패턴"을 만들어 목표 페르소나 분포를 마련했고, 실행 결과가 이 분포와 통계적으로 크게 어긋나지 않는지도 사후 검증했다.

## 3.6 보안 챌린지(VQA) 자동 풀이

보안 챌린지는 정답만 맞히면 되는 문제가 아니었다.
각 라운드에서 글러브를 목표 위치까지 드래그한 뒤, 타이밍 인디케이터가 특정 조건을 만족하는 정확한 순간에 놓아야 했다.
즉 위치 정확도와 시간 정확도를 동시에 만족해야 했다.

이 부분은 LLM이나 비전 모델이 아니라 결정론 규칙으로 풀었다.

| 기준 | 결정론 규칙 (화면 요소 좌표 기반) | LLM / 비전 모델 |
|------|----------------------------------|-----------------|
| 결정 지연 | ms 수준 | 수백 ms ~ 수 초 |
| 정확도 | 픽셀 단위 | 배경·해상도에 따라 흔들림 |
| 탐지 회피 | 브라우저 내 완결 | 외부 호출 흔적 가능 |
| 재현성 | 완전 결정론 | 동일 입력에도 출력 변동 가능 |

풀이 흐름은 단순하다.
1. 준비 상태를 확인하고 시작한다.
2. 글러브가 나타나면 자연스러운 경로로 목표 지점까지 드래그한다.
3. 타이밍 인디케이터가 조건을 만족하는 순간 놓는다.

여기에 몇 가지 보정 장치를 붙였다.

| 장치 | 원리 | 해결한 문제 |
|------|------|-------------|
| 타이밍 엄격화 | 상위 임계값을 한 번 이상 넘긴 기록이 있어야 놓기 허용 | 섣부른 놓기 |
| 드래그 후 재확인 | 실제 위치가 목표에서 멀면 재드래그 | 착지 오차 |
| 동시 풀이 제한 | 스웜 차원 직렬화 | 서버 과민 반응 |

즉 챌린지는 "정답 제출"이 아니라 브라우저 안에서 자연스럽고 정확하게 수행되는 실시간 조작 과제로 다뤘다.

## 3.7 스웜 구조 — 관측·결정·반영의 비동기 분리

스웜은 여러 에이전트를 한 PC 안에서 동시에 돌리되, LLM이 실행 흐름을 막지 않도록 관측·결정·반영을 분리했다.

| 역할 | 담당 | 특징 |
|------|------|------|
| 관측 | 이벤트 버스 | 에이전트가 블로킹 없이 이벤트 발행 |
| 결정 | LLM 코디네이터 | 필요할 때만 이벤트를 모아 전략 판단 |
| 반영 | 공유 전략 저장소 | 읽기는 즉시, 쓰기는 원자적으로 갱신 |

핵심은 에이전트가 LLM 응답을 기다리지 않는다는 점이다.
에이전트는 이벤트를 발행한 뒤 계속 진행하고, LLM은 뒤에서 전략만 업데이트한다.
결정이 필요한 순간 에이전트는 최신 전략을 읽기만 하면 된다.

이 때문에 읽기와 쓰기의 동기화도 비대칭으로 가져갔다.

| 연산 | 동기화 방식 | 이유 |
|------|-------------|------|
| 전략 읽기 | 잠금 없음 | 결정 지점에서 지연 최소화 |
| 전략 쓰기 | 잠금 사용 | 여러 필드 동시 업데이트의 원자성 확보 |

좌석 확보처럼 즉시 반응해야 하는 이벤트는 더 강하게 처리했다.

```
좌석 확보 성공 발생
    ↓
① 확보된 좌석을 다른 에이전트의 회피 목록에 즉시 추가
    ↓
② 그 다음에만 LLM이 넓은 전략 판단 수행
```

즉 LLM이 판단할 일과, 결정론으로 즉시 반영할 일을 분리해 충돌을 줄였다.

경량 LLM을 쓴 이유도 같다.
이 스웜의 판단은 "이 구역이 막혔으니 다른 구역으로 갈까" 수준이었기 때문에, 복잡한 추론보다는 빠른 응답과 낮은 비용이 더 중요했다.

## 3.8 공격 경로 — 좌석 추천 모드와 직접 좌석 선택 모드

공격은 두 가지 경로로 나뉜다.

| 구분 | 직접 좌석 선택 모드 | 좌석 추천 모드 |
|------|-------------------|---------------|
| 선택 주체 | 에이전트가 구역·좌석 직접 선택 | 시스템 자동 배정 |
| 자동화 난이도 | 높음 | 낮음 |
| 타깃 정확도 | 높음 | 낮음 |
| 주요 시나리오 | 프리미엄 구역 스나이핑 | 앞자리 대량 점유 |

직접 좌석 선택 모드에서는 에이전트끼리 담당 구역과 시작 열을 나눠 충돌을 줄였다.
반면 추천 모드에서는 시스템이 자동 배정하므로, 빠른 진입과 반복 시도 자체가 더 중요했다.

일부 스웜은 추천 모드로 넓게 점유하고, 다른 스웜은 특정 인기 구역을 직접 노리는 식으로 병행했다.
이렇게 해야 한 가지 방식만 시험하는 것이 아니라 다양한 공격 벡터를 동시에 재현할 수 있다.

## 3.9 분산 시뮬레이션

한 PC 안에서는 보통 5~10개의 에이전트를 돌렸고, 여러 사람이 각자 PC에서 스웜 하나씩 실행했다.
전체적으로는 최대 약 100명의 독립 에이전트가 동시에 들어오는 조건을 만들 수 있었다.

여기서 중요한 점은 PC 간 통신을 두지 않았다는 것이다.
실제 공격도 하나의 중앙 통제 봇넷처럼 움직이기보다, 서로 다른 IP와 기기에서 독립적으로 들어오는 경우가 많다.
따라서 이 실험의 목적은 "중앙집중식 완전 협업 공격"이 아니라, 여러 독립 공격원이 동시에 진입할 때 방어가 세션 단위 상관관계를 어떻게 처리하는지를 보는 데 있었다.

실행 흐름은 단순했다.

```
환경 준비 → 세션 파일 생성 → 스웜 실행 → 종료 시 지표 집계 → PC별 결과 통합
```

## 3.10 감사와 증거 — 모든 결정은 사후 재현 가능

이 시스템은 공격 성공 여부만 보는 도구가 아니라, 왜 그렇게 됐는지 나중에 다시 설명할 수 있는 도구여야 했다.
그래서 결과를 세 층으로 남겼다.

| 산출물 | 형식 | 용도 |
|--------|------|------|
| 구조화 이벤트 로그 | JSON Lines | 흐름 분석, 지표 산출 |
| 상태 스냅샷 | PNG / HAR / 콘솔 로그 | 당시 UI·네트워크 맥락 확인 |
| 자동 집계 지표 | 요약 파일 | 퍼널, 성공률, 챌린지 결과 요약 |

이 셋은 서로 보완 관계다.
이벤트 로그만 있으면 화면 상황을 모르고, 스냅샷만 있으면 전체 흐름을 빠르게 보기 어렵고, 요약 지표만 있으면 개별 실패 원인을 설명할 수 없다.
따라서 공격 에이전트는 단순 실행기가 아니라 사후 분석 가능한 실험 장치로 설계했다.

## 3.11 관찰된 실패 패턴과 대응

실제 실행에서는 단순한 성공·실패보다 실패 원인을 줄여 가는 과정이 중요했다.

| # | 실패 유형 | 발생 상황 | 대응 |
|---|-----------|-----------|------|
| 1 | 큐 미개방 충돌 (409) | 정각 클릭했지만 서버 큐가 아직 열리지 않음 | 500ms 간격 반복 클릭 |
| 2 | 챌린지 타이밍 실패 | 조건 진입 직후 너무 빨리 놓음 | 상위 임계값 통과 기록 필요 |
| 3 | 챌린지 착지 실패 | 드래그 위치가 목표에서 벗어남 | 실제 위치 재확인 후 재드래그 |
| 4 | 챌린지 통과 후 구역 화면 정체 | 이미 설정된 인원수를 다시 설정 시도 | 현재 값 확인 후 스킵 |
| 5 | 좌석 충돌 (409) | 여러 에이전트가 같은 좌석 시도 | 공유 점유 리스트 + 시작 열 분산 |
| 6 | 장거리 이동 시 속도 저하 | 긴 곡선 이동 누적으로 지연 | 사전 위치 선점 후 짧은 보정 이동 |
| 7 | 구역 선택 후 빈 시간 | 서버 응답·프론트 렌더 대기 누적 | 버퍼와 폴링 주기 단축 |
| 8 | 방어 탐지 (403) | 방어가 실제로 차단 | 차단율 자체를 성공 지표로 반영 |

여기서 중요한 것은 공격 측 성공 기준을 다시 정의한 점이다.
목표는 "항상 뚫는다"가 아니라, 어느 공격 수준에서 어느 정도 차단되는지 측정한다는 데 있다.
따라서 고수준 공격에서조차 차단율이 높게 나오면, 그것은 공격 실패가 아니라 방어 강도의 증거다.

운영 중 추가로 손본 부분도 있었다.

| 문제 | 관찰 | 수정 |
|------|------|------|
| 결제 페이지 도달 후 브라우저 자동 종료 | 사용자가 결제 입력 전 창 종료 | DONE 시 브라우저 유지 |
| Ctrl+C 시 요약 파일 손실 | 마지막 저장이 중도 종료에 휘말림 | 종료 전 미리 저장 |
| 구역 적중률 지표 0% | 이벤트에 실제 구역 정보 없음 | 구역명 함께 기록 |
| 추천 모드에서 무의미한 구역 지시 | 시스템 자동 배정인데 LLM이 구역 변경 지시 | 집계에서 자동 제외 |

## 3.12 다섯 문장 요약 — 발표 핵심 메시지

1. 공격 에이전트는 방어 내부를 모르는 블랙박스 전제에서 설계했다.
2. 단일 에이전트는 결정론적 상태머신으로 만들고, 스웜 전략에만 LLM을 제한적으로 사용했다.
3. 마우스는 "형태는 인간, 속도는 장인" 원칙으로 합성해 탐지 회피와 경쟁력을 동시에 노렸다.
4. 스웜은 관측→결정→반영을 비동기로 분리해 LLM 지연이 실제 실행 흐름을 막지 않도록 했다.
5. 모든 실행은 로그, 스냅샷, 지표로 남겨 사후 재현 가능한 검증 실험이 되도록 만들었다.
