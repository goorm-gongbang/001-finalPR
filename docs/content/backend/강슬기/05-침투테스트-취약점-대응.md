### 2.1 [CRITICAL] C-09: 좌석 IDOR — 1인당 Hold 수량 무제한

**취약점 설명**

좌석 Hold API에 수량 제한이 없어서, 공격자가 한 번에 수십~수백 석을 Hold하여 좌석 점유 DoS 공격이 가능했다.

```
# 공격 시나리오
POST /seat/matches/{matchId}/seat-holds
Body: { seatIds: [1, 2, 3, ... 100] }  ← 100석 한번에 Hold 가능!
```

**대응 (패치 완료)**

- 1인당 Hold 수량을 **최대 8석**으로 제한
- `seatIds.size()` 사전 검증 로직 추가
- 경기당 최대 예매 제한도 8매로 설정 (PAYMENT_PENDING, PAID, UNDER_REVIEW 포함)
- 파일: `SeatHoldTransactionalService.java`

---

### 2.2 [HIGH] H-04: Gateway 쓰기 인증 우회

**취약점 설명**

API-Gateway의 JWT 인증 필터가 **경로만 체크하고 HTTP 메서드를 무시**했다. 공개 경로(GET)에 POST/PUT/DELETE를 보내면 인증 없이 쓰기 작업이 가능했다.

```
# 공격 시나리오
POST /queue/matches/1/status  ← GET만 공개인데 POST도 통과!
```

**대응 (패치 완료)**

- `WhitelistEntry`를 HTTP 메서드 + 경로 구조체로 변경
- 공개 API는 GET만 허용, 쓰기 메서드는 반드시 인증 필요
- 파일: `JwtAuthenticationFilter.java`

---

### 2.3 [HIGH] H-05: Actuator 전체 노출

**취약점 설명**

Spring Boot Actuator의 모든 엔드포인트(`/actuator/metrics`, `/actuator/env`, `/actuator/beans` 등)가 인증 없이 접근 가능했다. 환경변수, 메트릭, 빈 정보 등 서버 내부 정보가 노출되었다.

```
# 공격 시나리오
GET /actuator/env  ← DB 비밀번호, JWT 키 등 노출 가능!
GET /actuator/metrics  ← 내부 메트릭 정보 노출
```

**대응 (패치 완료)**

- staging/prod 환경에서 `health`, `prometheus`만 허용
- 나머지 Actuator 엔드포인트는 `denyAll`
- `@Profile` 분리로 로컬 개발 환경에서는 전체 접근 가능 유지
- 파일: `ActuatorSecurityConfig.java` (각 모듈)

---

### 2.4 [HIGH] H-09: Thread Pool 무제한

**취약점 설명**

`TaskExecutor`의 max 값이 `Integer.MAX_VALUE`(약 21억)로 설정되어 있었다. 대량 요청으로 스레드를 무한 생성시켜 OOM(Out of Memory) DoS 공격이 가능했다.

**대응 (패치 완료)**

- Tomcat max-threads: **200**
- Spring TaskExecutor: core 8, max **50**, queue 100
- HikariCP max-pool-size: **20**
- staging/prod `application.yaml`에 명시적 제한 설정
- 파일: `application-staging.yaml` (각 모듈)

---

### 2.5 [MEDIUM] M-09: 주문 ID 순차 열거

**취약점 설명**

주문 ID가 auto-increment Long ID만 사용되어, 순차적으로 ID를 증가시키며 다른 유저의 주문 존재 여부를 확인할 수 있었다. 또한 403(Forbidden)과 404(Not Found) 응답 차이로 리소스 존재 여부가 노출되었다.

```
# 공격 시나리오
GET /order/mypage/orders/1    ← 404 (없음)
GET /order/mypage/orders/2    ← 403 (있지만 내 것 아님) ← 존재 확인!
GET /order/mypage/orders/3    ← 200 (내 것)
```

**대응 (패치 완료)**

- `orderNumber`(UUID) 필드 추가: `ORD-{UUID}` 형식
- 소유권 불일치 시 403 대신 **404 반환** (존재 여부 노출 방지)
- `[Security]` 워닝 로그 추가 (비정상 접근 추적)
- 파일: `Order.java`, `PaymentService.java`, `MyPageTicketService.java`

---

### 2.6 [MEDIUM] M-12: Rate Limiting 미설정

**취약점 설명**

모든 API에 Rate Limiting이 없어서 브루트포스 공격(로그인 무한 시도), 대기열 폴링 폭격 등이 가능했다.

**대응 (패치 완료)**

- **prod 환경** Redis INCR + TTL 60초 기반 Rate Limiting 적용
  - 로그인 엔드포인트: IP당 **10회/분**
  - 일반 API 엔드포인트: IP당 **100회/분**
- 대기열 진입(`/queue/matches/*/enter`): 유저당 **3 토큰/초**, 버스트 10
- Redis 장애 시 Fail-open (가용성 우선)
- staging은 부하테스트용으로 해제
- 파일: `RateLimitingFilter.java`
