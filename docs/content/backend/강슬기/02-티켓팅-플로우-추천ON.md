## STEP 1: 경기 상세 페이지

**사용자가 보는 화면**
- 추천 ON/OFF 토글
- 인접석/비인접석 ON/OFF 토글
- 인원수 선택 (최대 8명)
- [예매하기] 버튼

**백엔드에서 하는 일: 예매 조건 저장**

1. 사용자가 설정한 예매 옵션(추천 ON, 인원수 N명, 인접석 토글)을 Seat 서비스의 Redis에 저장
   - 키: `seat:booking-options:{matchId}:{userId}`
   - TTL: 15분 (900초)
2. Queue 전용 Redis에 PreQueue 마커 동기화
   - 키: `queue:precheck:booking-option:{matchId}:{userId}`
   - 이유: 대기열 진입 시 예매 옵션 설정 여부를 빠르게 검증하기 위함
3. Resilience4j @Retry로 마커 동기화 3회 재시도 (실패 시 booking-options도 롤백)

```
POST /seat/matches/{matchId}/booking-options
Body: { recommendationEnabled: true, ticketCount: 4, nearAdjacentToggle: false }

→ Seat Redis: SET seat:booking-options:{matchId}:{userId} (TTL 900s)
→ Queue Redis: SET queue:precheck:booking-option:{matchId}:{userId} (TTL 900s)
```

## STEP 2: 대기열 화면

**사용자가 보는 화면**
- 내 순번: 12,240번
- 앞에 6,000명이 대기 중
- 예상 대기 시간

**백엔드에서 하는 일: 대기열 시스템**

### 대기열 진입
1. Redis Sorted Set에 순번 등록
   - 키: `queue:wait:{matchId}`
   - Score: 진입 시각(밀리초) → 선입선출 보장
2. 스케줄러가 1초마다 100명씩 Ready 상태로 승격
3. 승격된 유저에게 **Admission Token** (RSA JWT) 발급
   - Claims: userId, matchId, type=ADMISSION
   - TTL: 15분
   - HttpOnly Cookie로 전달
4. HttpOnly + Secure + SameSite=None 쿠키 설정

### 대기열 폴링 (동적 간격)

| 대기 순위 | 폴링 간격 | 이유 |
|----------|----------|------|
| ≤ 100 | 1.5초 | 곧 입장할 유저에게 빠른 피드백 |
| ≤ 1,000 | 3초 | 중간 대기 |
| > 1,000 | 5초 | 긴 대기, 서버 부하 절감 |
| READY 상태 | 1초 | 입장 가능 알림 |

```
GET /queue/matches/{matchId}/status
→ { status: "WAITING", rank: 234, totalWaiting: 5000, pollingMs: 1500 }
→ { status: "READY", admissionToken: "eyJ...", pollingMs: 1000 }
```

### Lazy 판정 (11시 정각 즉시 입장)
- 기존: DB의 `saleStatus == ON_SALE` 확인 → 스케줄러 지연 1~2분
- 현재: `now >= salesOpenAt` 시간 비교 (Lazy) → 정확히 11:00:00에 입장 가능
- 스케줄러는 10:59에 미리 실행하여 DB 상태 전환

## STEP 3: 공잡기 미션 (VQA)

**사용자가 보는 화면**
- 야구 잡기 게임 VQA 실행
- 미션 성공 시 다음 단계로

**AI 서버가 뒤에서 하는 일: 봇 탐지 · 방어**

```
┌─────────────────────────────────────────────────────────────────┐
│                    AI 방어 체계 (뒤에서 동작)                      │
│                                                                 │
│  1. 행동 데이터 수집 시작                                         │
│     경기 상세 페이지 → 예매하기 10초 전(의심)                       │
│     마우스 움직임 · 클릭 패턴 · 체류 시간 수집 시작                  │
│                                                                 │
│  2. Turnstile 검증 + 봇 사전 차단                                │
│     대기열 진입 시 Cloudflare Turnstile 검증                      │
│     API precheck 결과 저장                                      │
│     실패 시 대기열 진입 자체 차단                                  │
│                                                                 │
│  3. 실시간 미들 점수 수집                                        │
│     T0~T1 (사람): 취 통과 (Pass)                                │
│     T2 (의심): 응답 지연 (연장 모니터링)                           │
│     T3 (봇): 즉시 차단 (403)                                    │
│                                                                 │
│  4. 좌석 선택 구간 감시 (cost_autha)                              │
│     좌석 선택/해제 API 호출 시 패턴 수집                           │
│     Envoy가 AI에게 미러링                                       │
│     일정 시 챌린지 요구 or 차단                                   │
│                                                                 │
│  5. 텔레메트리 수집 → 벡터코드                                    │
│     판정 결과와 페이지 이벤트를 250ms 누적                         │
│     → 주기적으로 S3(Object Storage)에 업로드                      │
│     → 사후 분석/비정상 로그 추적                                  │
│                                                                 │
│  6. 사후 분석 (오프라인 AI)                                      │
│     결제 후에도 묶음 데이터 딥러닝 분석                             │
│     의심·은밀 패턴 탐지 → 주문 검토(UNDER_REVIEW)                  │
│     매크로 짙은 후보를 백엔드에 전달                                │
└─────────────────────────────────────────────────────────────────┘
```

**백엔드가 제공하는 차단 API**
- `POST /internal/users/{userId}/block` (X-Internal-Api-Key 필수)
- 차단 시 User.status → BLOCKED → Kafka `user-blocked` 이벤트 → Order-Core에서 주문 UNDER_REVIEW 처리

## STEP 4: 추천 블록 카드 선택

**사용자가 보는 화면**
- 오렌지석 20블록 · 옐로석 15개
- 블루석 30블록 · 레드석 12개
- 퍼플석 10블록 · 핑크석 12개
- 최상 위 1위 ~ 10위까지 블록 카드
- 각 블록당 가용 연석 수 표시

**백엔드에서 하는 일: 추천 알고리즘 계산**

1. 온보딩에서 입력받은 유저 선호 데이터 조회
   - 선호 구단 (1루/3루 자동 판정)
   - 뷰포인트 우선순위 (1~3순위)
   - 응원석 인접 선호 (NEAR/FAR/ANY)
   - 선호 블럭 목록
2. 선호 블럭별 가용 연석 수 계산
   - 실연석(같은 열 연속 N석) 카운트
   - 준연석(2열 겹침 N석) 카운트 (인접석 토글 ON 시)
3. 블럭 랭킹 결정
   - 1차: 가용 연석 수 (차이가 10석 이내면 선호도 점수로 타이브레이킹)
   - 2차: 선호도 점수 (뷰포인트 0~30 + 구단 0~25 + 응원석 0~15 = 최대 70점)
4. 상위 10개 블럭을 1순위~10순위로 반환

```
GET /seat/matches/{matchId}/recommendations/blocks
→ [
    { rank: 1, blockId: 204, blockName: "오렌지 C", consecutiveCount: 42, score: 65 },
    { rank: 2, blockId: 207, blockName: "옐로 A", consecutiveCount: 38, score: 55 },
    ...
  ]
```

## STEP 5: 좌석 자동 배정 완료

**사용자가 보는 화면**
- 자동 배정된 좌석 정보 (예: 204블록 3열 1~4번)
- 5분 내 결제 필요
- 만료 시 자동 해제

**백엔드에서 하는 일: 좌석 자동 배정 + Hold**

1. **블럭 단위 Redisson 분산 락 획득** (Wait 3초, Watch Dog 자동 갱신)
   - 키: `seat:recommendation:match:{matchId}:block:{blockId}`
2. 기존 Hold가 있다면 삭제 + 좌석 AVAILABLE 복원
3. **실연석 탐색** (최대 3회 재시도)
   - 같은 열에서 연속 N석 탐색
   - 우선순위: 앞줄 → 통로 근접 → 왼쪽
   - 조건부 UPDATE: `markBlockedIfAvailable()` → return 0이면 충돌 → 롤백 → 재시도
4. 실연석 실패 시 **준연석 폴백** (인접석 토글 ON)
   - 인접 2열에 걸친 N석 탐색
   - 우선순위: 앞줄 합 → 겹침 수 최대 → 통로 근접
5. **SeatHold 생성** (TTL 5분)
   - MatchSeat 상태: AVAILABLE → BLOCKED
   - SeatHold 레코드: userId, matchSeatId, expiresAt
6. **분산 락 해제** (트랜잭션 커밋 후)

```
POST /seat/matches/{matchId}/recommendations/blocks/{blockId}/assign
→ { holdId: 42, seats: [{row: 3, col: 1}, {row: 3, col: 2}, ...], expiresAt: "..." }
```

## STEP 6: 주문서 확인

**사용자가 보는 화면**
- 경기 정보, 좌석 정보, 가격
- 좌석 금액 + 예매 수수료 2,000원
- 결제 수단 선택 (카카오페이 / 토스페이 / 무통장입금)

**백엔드에서 하는 일: 주문 생성**

1. Hold 만료 여부 검증
2. 좌석 가격 서버사이드 재계산 (클라이언트 조작 방지)
3. 기존 미결제 주문 자동 취소 (같은 경기 같은 유저)
4. Order 엔티티 생성 (상태: PAYMENT_PENDING)
5. OrderSeat 엔티티 생성 (좌석별 가격/블럭/섹션 정보)

```
POST /order/mypage/orders
→ { orderId: 123, orderNumber: "ORD-a1b2c3d4", totalPrice: 42000 }
```

## STEP 7: 결제 완료

**사용자가 보는 화면**
- 결제 완료 화면
- 예매 내역 확인 가능

**백엔드에서 하는 일: 결제 처리 + 좌석 확정**

1. **결제 처리**
   - 카카오페이/토스페이: 즉시 PAID 전환 (Mock)
   - 무통장입금: 입금 기한 설정 (다음날 23:59 or 경기 시작 3시간 전 중 이른 시각)
2. **Kafka 이벤트 발행**: `payment-completed` (matchSeatIds 포함)
3. **Seat 서비스에서 좌석 상태 전환**: BLOCKED → SOLD
4. Hold 해제 (좌석이 SOLD로 확정되었으므로)

```
결제 완료 후 이벤트 흐름:
Order-Core ──[payment-completed]──→ Kafka ──→ Seat (BLOCKED → SOLD)
```

## 토큰으로 단계가 이어짐

```
Access Token (로그인)
     │
     ├→ 대기열 진입 → Admission Token 발급 (대기열 → 좌석 진입권)
     │                    │
     │                    ├→ 좌석 Hold (5분 TTL)
     │                    │       │
     │                    │       ├→ 주문 생성 (PAYMENT_PENDING)
     │                    │       │       │
     │                    │       │       └→ 결제 → SOLD
     │                    │       │
     │                    │       └→ 5분 초과 시 자동 해제
     │                    │
     │                    └→ 15분 초과 시 Admission Token 만료
     │
     └→ Refresh Token → Access Token 재발급 (RTR 방식)
```

## 데이터 저장소별 역할

| 저장소 | 용도 | 키/테이블 |
|-------|------|----------|
| **Redis (Seat)** | 예매 조건 캐시 | `seat:booking-options:{matchId}:{userId}` |
| **Redis (Queue)** | 대기열 ZSET | `queue:wait:{matchId}` |
| **Redis (Queue)** | 세션 · VQA 상태 | `queue:ready:{matchId}:{userId}` |
| **Redis (Lock)** | Redisson 분산 락 | `seat:recommendation:match:*:block:*` |
| **PostgreSQL** | 좌석 상태 | `match_seats` (AVAILABLE/BLOCKED/SOLD) |
| **PostgreSQL** | 주문 · 결제 | `orders`, `payments` |
| **Kafka** | 결제→좌석 확정 | `payment-completed` 토픽 |
