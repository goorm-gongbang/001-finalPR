# PlayBall 추천 서비스 알고리즘

> 유저 선호도 기반 블럭 추천 + 좌석 자동 배정 알고리즘 상세 설명

---

## 1. 추천 서비스 개요

PlayBall의 추천 서비스는 일반 티켓팅 사이트와 차별화된 핵심 기능이다. 유저가 직접 좌석맵에서 좌석을 고르는 대신, **온보딩 시 입력한 선호 데이터**를 기반으로 최적의 블럭을 1순위~10순위로 추천하고, 블럭을 선택하면 **서버가 자동으로 최적 연석을 배정**한다.

```
┌──────────────────────────────────────────────────────────────────┐
│                    추천 서비스 전체 흐름                            │
│                                                                  │
│  1. 온보딩 선호 데이터 수집 (회원가입 직후)                          │
│     ├─ 선호 구단 (예: 삼성 라이온즈)                                │
│     ├─ 뷰포인트 우선순위 (1~3순위: CENTER, INFIELD_1B, ...)         │
│     ├─ 응원석 인접 선호 (NEAR / FAR / ANY)                         │
│     └─ 선호 블럭 목록 (사용자가 직접 선택)                           │
│                                                                  │
│  2. 블럭 추천 (예매 시)                                            │
│     ├─ 선호 블럭 대상으로 가용 연석 수 계산                          │
│     ├─ 선호도 점수(0~70) 계산                                     │
│     └─ 1순위~10순위 랭킹 반환                                     │
│                                                                  │
│  3. 좌석 자동 배정 (블럭 선택 시)                                   │
│     ├─ 분산 락 획득 (블럭 단위)                                    │
│     ├─ 실연석(같은 열 N석) 우선 탐색                               │
│     ├─ 준연석(2열 겹침 N석) 폴백                                   │
│     └─ Hold 생성 (5분 TTL)                                       │
└──────────────────────────────────────────────────────────────────┘
```

## 2. 온보딩 선호 데이터

회원가입 직후 온보딩 과정에서 아래 데이터를 수집한다.

### 2.1 수집 데이터

| 데이터 | 엔티티 | 설명 | 추천 알고리즘 사용 |
|-------|--------|------|-----------------|
| 선호 구단 | `OnboardingPreference.favoriteClub` | 예: 삼성 라이온즈 | 1루/3루 블럭 방향 결정 |
| 응원석 인접 선호 | `OnboardingPreference.cheerProximityPref` | NEAR / FAR / ANY | 응원석 근접 블럭 가산점 |
| 뷰포인트 우선순위 | `OnboardingViewpointPriority` | 1~3순위 (CENTER, INFIELD_1B, ...) | 뷰포인트 매칭 가산점 |
| 선호 블럭 목록 | `OnboardingPreferredBlock` | 유저가 직접 선택한 블럭들 | 추천 대상 블럭 필터링 |

### 2.2 구단 기반 1루/3루 자동 판정

```
경기: 삼성 라이온즈 vs 한화 이글스
유저 선호 구단: 삼성 라이온즈

Case 1: 삼성이 홈(1루)일 때
  → 1루(홈) 블럭 쪽 추천 가산점 +25점

Case 2: 삼성이 어웨이(3루)일 때
  → 3루(어웨이) 블럭 쪽 추천 가산점 +25점

Case 3: 삼성이 참여하지 않는 경기 (예: LG vs 두산)
  → 온보딩에서 입력받은 뷰포인트 우선순위로 판정
  → 1루/3루/외야 중 선호하는 방향을 우선시
```

## 3. 블럭 추천 알고리즘

### 3.1 추천 대상 필터링

추천 대상은 유저의 **선호 블럭 목록**(`OnboardingPreferredBlock`)으로 한정한다. 선호 블럭이 아닌 블럭은 아무리 좌석이 많아도 추천하지 않는다.

### 3.2 가용 연석 수 계산

각 선호 블럭에 대해 현재 AVAILABLE 상태인 좌석을 조회하고, 요청 인원수(N)에 대한 연석 조합 수를 계산한다.

**실연석 (Real Consecutive)**: 같은 열에서 연속 N석

```
예: 3열에 1,2,3,4,5,8,9,10번이 AVAILABLE이고 N=4이면
    연속 구간: [1,2,3,4,5] → 4연석 조합: [1-4], [2-5] = 2개
             [8,9,10] → 3석이라 4연석 불가
    실연석 카운트 = 2
```

**준연석 (Semi-Consecutive)**: 인접 2열에 걸친 N석 (수평 겹침 있음)

```
예: 3열에 [1,2,3], 4열에 [2,3,4,5]가 AVAILABLE이고 N=4이면
    3열에서 2석 + 4열에서 2석 → 수평 겹침 확인
    겹침 조건: 두 그룹의 열 범위가 겹쳐야 함 (바로 위아래)
    준연석 카운트 = 겹침이 있는 조합 수
```

### 3.3 선호도 점수 (PreferenceScore)

블럭마다 0~70점의 선호도 점수를 계산한다.

```
총점 = 뷰포인트 점수(0~30) + 구단 선호 점수(0~25) + 응원석 인접 점수(0~15)
```

#### 뷰포인트 점수 (0~30점)

유저의 뷰포인트 우선순위(1~3순위)와 블럭의 위치를 매칭한다.

| 매칭 순위 | 점수 |
|----------|------|
| 1순위 매칭 | **30점** |
| 2순위 매칭 | **20점** |
| 3순위 매칭 | **10점** |
| 매칭 없음 | 0점 |

뷰포인트 종류: `CENTER`, `INFIELD_1B`, `INFIELD_3B`, `OUTFIELD_L`, `OUTFIELD_C`, `OUTFIELD_R`

#### 구단 선호 점수 (0~25점)

| 조건 | 점수 |
|------|------|
| 유저의 선호 구단이 해당 블럭의 홈/어웨이 진영과 일치 | **25점** |
| 그 외 | 0점 |

예시:
- 삼성 팬 + 삼성 홈 경기 + 1루 블럭 → 25점
- 삼성 팬 + 삼성 원정 경기 + 3루 블럭 → 25점
- 삼성 팬 + 삼성 미참여 경기 → 0점 (뷰포인트 점수로 판정)

#### 응원석 인접 점수 (0~15점)

각 블럭은 `cheerRank` 값을 가진다 (응원석까지의 거리 순위).

| 유저 선호 | 조건 | 점수 |
|----------|------|------|
| NEAR (응원석 인접 선호) | cheerRank ≤ 3 | **15점** |
| NEAR | cheerRank > 3 | 0점 |
| FAR (응원석 비인접 선호) | cheerRank > 3 | **15점** |
| FAR | cheerRank ≤ 3 | 0점 |
| ANY | - | 0점 |

### 3.4 블럭 랭킹 알고리즘

블럭을 정렬하여 1순위~10순위를 결정한다.

```
정렬 기준:

1차: 가용 연석 수 (내림차순)
     단, 두 블럭의 연석 수 차이가 10석 이내이면 → 2차 기준으로 판정

2차: 선호도 점수 (내림차순)
     점수가 같으면 → 연석 수로 최종 결정

CONSECUTIVE_COUNT_THRESHOLD = 10석
```

```
예시 (N=4명 요청):

블럭 A: 연석 42개, 선호도 65점
블럭 B: 연석 38개, 선호도 55점
블럭 C: 연석 40개, 선호도 70점

A vs C: |42-40| = 2 ≤ 10 → 선호도 비교 → C(70) > A(65) → C가 상위
A vs B: |42-38| = 4 ≤ 10 → 선호도 비교 → A(65) > B(55) → A가 상위
최종: C(1순위) → A(2순위) → B(3순위)

만약 블럭 D: 연석 5개, 선호도 70점이라면
A vs D: |42-5| = 37 > 10 → 연석 수 비교 → A(42) > D(5) → A가 상위
(선호도가 아무리 높아도 좌석이 너무 적으면 하위 랭크)
```

## 4. 좌석 자동 배정 알고리즘

유저가 추천 블럭을 선택하면, 서버가 해당 블럭 내에서 최적의 N연석을 자동 배정한다.

### 4.1 전체 배정 흐름

```
┌─────────────────────────────────────────────────┐
│            좌석 자동 배정 흐름                     │
│                                                 │
│  1. Redisson 블럭 분산 락 획득                    │
│     키: seat:recommendation:match:{m}:block:{b}  │
│     Wait: 3초, Watch Dog 자동 갱신               │
│                                                 │
│  2. 기존 Hold 정리                               │
│     이전 Hold 삭제 + 좌석 AVAILABLE 복원           │
│                                                 │
│  3. 실연석 탐색 (최대 3회 재시도)                   │
│     ├─ 성공 → Hold 생성 → 완료                   │
│     └─ 실패(충돌) → 다른 구간으로 재시도            │
│                                                 │
│  4. 준연석 폴백 (인접석 토글 ON일 때만)             │
│     ├─ 성공 → Hold 생성 → 완료                   │
│     └─ 실패 → 에러 응답                          │
│                                                 │
│  5. 분산 락 해제 (트랜잭션 커밋 후)                 │
└─────────────────────────────────────────────────┘
```

### 4.2 실연석 탐색 (RealConsecutiveFinder)

같은 열에서 연속 N석을 찾는다.

```
알고리즘:
1. 블럭 내 AVAILABLE 좌석 전체 조회
2. 열(row)별로 그룹핑
3. 각 열에서 연속 구간(segment) 추출
4. N석 이상인 구간에서 슬라이딩 윈도우로 후보 생성
5. 후보 정렬:
   - Priority 1: 앞줄 (rowNo 오름차순)
   - Priority 2: 통로 근접 (aisleDistance 오름차순)
   - Priority 3: 왼쪽 (startCol 오름차순)
```

**통로 근접도 (Aisle Distance) 계산**

```
rowStartCol = 해당 열의 첫 번째 좌석 열번호
rowEndCol = 해당 열의 마지막 좌석 열번호

leftDistance = groupStartCol - rowStartCol   (왼쪽 통로까지 거리)
rightDistance = rowEndCol - groupEndCol       (오른쪽 통로까지 거리)

aisleDistance = min(leftDistance, rightDistance)

→ 통로 선호 유저: aisleDistance가 작은 좌석 우선
→ 중앙 선호 유저: aisleDistance가 큰 좌석 우선 (구현상 통로 근접 우선)
```

### 4.3 준연석 탐색 (SemiConsecutiveFinder)

인접 2열에 걸친 N석을 찾는다. 실연석을 찾을 수 없을 때의 폴백이다.

```
알고리즘:
1. 열 번호 정렬 후 인접 열 쌍 순회
2. 두 열이 물리적으로 인접해야 함 (rowNo 차이 = 1)
3. 상위 열에서 K석 + 하위 열에서 (N-K)석 조합
4. 수평 겹침(overlap) 검증:
   overlap = min(upperEnd, lowerEnd) - max(upperStart, lowerStart) + 1
   overlap > 0 이어야 유효 (바로 위아래에서 겹쳐야 함)
5. 후보 정렬:
   - Priority 1: 앞줄 합 (rowSum 오름차순)
   - Priority 2: 겹침 수 최대 (overlapCount 내림차순)
   - Priority 3: 통로 근접 (avgAisleDistance 오름차순)
```

```
예시 (N=4, 인접석 토글 ON):

3열: [1, 2, 3]     ← AVAILABLE
4열: [2, 3, 4, 5]  ← AVAILABLE

후보: 3열에서 2석(2,3) + 4열에서 2석(2,3) → overlap=2 ✅
     3열에서 1석(3)   + 4열에서 3석(3,4,5) → overlap=1 ✅
     3열에서 3석(1,2,3) + 4열에서 1석(2) → overlap=1 ✅
```

### 4.4 좌석 Hold (markBlockedIfAvailable)

좌석 배정 시 조건부 UPDATE로 동시성 충돌을 감지한다.

```sql
UPDATE match_seats
SET sale_status = 'BLOCKED'
WHERE id = :matchSeatId
  AND sale_status = 'AVAILABLE'
-- return 0이면 이미 다른 유저가 선점 → 롤백 후 재시도
```

- 성공: 모든 좌석 BLOCKED + SeatHold 레코드 생성 (TTL 5분)
- 실패 (return 0): 이미 BLOCKED/SOLD된 좌석 발견 → 지금까지 BLOCKED한 좌석 롤백 → 다른 연석 구간으로 재시도
- 최대 재시도: 실연석 3회 + 준연석 3회

## 5. 추천 OFF (일반 선택) 모드

추천을 사용하지 않는 유저는 좌석맵에서 직접 좌석을 클릭하여 선택한다.

| 항목 | 추천 ON | 추천 OFF (일반 선택) |
|------|---------|-------------------|
| 블럭 선택 | 서버가 추천 (1~10순위) | 유저가 직접 선택 |
| 좌석 배정 | 서버가 자동 배정 (연석) | 유저가 좌석맵에서 직접 클릭 |
| 락 단위 | 블럭 단위 Redisson RLock | 좌석 단위 Redisson RLock |
| 인원수 | 예매 옵션에서 사전 설정 | 클릭한 좌석 수 = 인원수 |
| API | `/recommendations/blocks/{blockId}/assign` | `/seat-holds` |

| `ticketing_seat_lock_wait_seconds` | Timer | 분산 락 대기 시간 |
| `ticketing_seat_hold_attempt_total` | Counter | Hold 시도 수 (mode=RECOMMEND/MAP) |
| `ticketing_seat_hold_success_total` | Counter | Hold 성공 수 |
| `ticketing_seat_recommend_degrade_total` | Counter | 준연석 폴백 발생 수 |
